Analysis of Intraday Stock Prices¶

David Chuang

Introduction¶

In the fast-paced world of financial markets, every second counts. The ability to predict and analyze stock price movements within a single trading day, known as intraday trading, can be the difference between profit and loss for traders and investors. This project aims to leverage the power of data science to delve into the intricacies of intraday stock price movements.

Our objective is to harness the potential of machine learning algorithms and statistical models to predict intraday stock price movements. We will be analyzing high-frequency data, which includes stock prices at different intervals within the trading day. This data is rich with insights and presents a unique set of challenges and opportunities for data scientists.

The project will involve data collection, preprocessing, exploratory data analysis (EDA), feature engineering, and model development. We will be using various data science tools and libraries to handle the large volume of intraday data and extract meaningful patterns from it.

By the end of this project, we aim to develop a robust model that can accurately predict the direction of intraday stock price movements. Predicting the direction of stock price movement could be valuable to traders.

Pros/Cons of High-Frequency Data¶

By analyzing higher-frequency intraday data, we can more quickly capture the reaction of the market to news or events.

However, high-frequency data has a few disadvantages, namely handling larger volumes of data and dealing with noise. A great deal of the variation in high-frequency price data can be explained by random variation, so there might be difficulty in extracting trends.

Data Collection¶

For this project we will be using AlphaVantage and analyzing SPY, which is an ETF that tracks the S&P 500 Index. We will get intraday stock price data from the past 12 months at an interval of 5 minutes.

Retrieving data from AlphaVantage's REST API¶

The below dictionary will be used as URL parameters to retrieve CSV files from AlphaVantage's REST API directly using pandas.

What is a REST API? In a nutshell, it's something that can be accessed from the web using HTTP requests. They take parameters either as part of a JSON request sent by a library or encoded in the URL. We will use a library to convert the dictionary of parameters below into a URL to request a CSV from AlphaVantage's website.

For more details about the API parameters and an example of a request URL, see the AlphaVantage API documentation. A basic description of each parameter we use is listed below.

  • apikey: Required to download data from AlphaVantage's API.
  • symbol: Stock market symbol to retrieve. This should be SPY.
  • datatype: Datatype to retrieve. We want CSVs so we can load as DataFrame easily.
  • function: What type of data to retrieve. TIME_SERIES_INTRADAY corresponds to intraday price data.
  • months: List of months to retrieve in YYYY-MM format.
    !! This is not an actual URL parameter. AlphaVantage allows you to retrieve one month at a time using the month parameter, so this list will be used in the code below to download one CSV for each month.
  • interval: The time interval between data points in our dataset.
  • outputsize: If this is not set to full, AlphaVantage will only output the past 100 data points.
  • extended_hours: Whether or not to include extended trading hours. During extended trading hours, the market behaves very differently, so we want to leave it out.

Downloading data as CSV using Pandas¶

The below function is used to download a CSV from a url using Pandas and save it as a file (or to skip if it already exists). This is helpful because a free AlphaVantage API key rate-limits you to 5 requests per minute. The function also renames 'time' to 'timestamp'.

The function takes as input the requested URL and file name to cache the data at.

  1. First, we check if the file already exists with os.path.exists. We store files at csv/[month].csv, where month is the YYYY-MM month parameter. If the file exists, we skip the below steps and just read from the cached file.
  2. If the file doesn't exist, the Pandas function pd.read_csv(url) can be used to read a CSV file directly from the Internet into a DataFrame.
  3. We then check if 'close' is in the list of columns after reading the data into a DataFrame. If not, then some error has occurred, either due to an error in our URL parameters or due to exceeding the rate limit. Raise an exception so we are notified and can try again later.
  4. If the DataFrame has been received properly and contains the data we want, we can go on to cache it as a CSV.
  5. Make sure the 'csv' directory exists with os.path.isdir('csv') and if not, create it with os.mkdir('csv').
  6. Then, save the DataFrame to file as CSV at the corresponding file path with df.to_csv(file_path, index=False). index=False is necessary because otherwise, Pandas will create an unnamed column for the index which serves no purpose to us.

# DataFrame manipulation
import pandas as pd

# Cache files
import os

def download_csv(url: str, file_name: str):
    file_path = f"csv/{file_name}.csv"
    
    # timestamp,open,high,low,close,volume
    if not os.path.exists(file_path):
        # print('Loading data from AlphaVantage url : %s' % url)
        df = pd.read_csv(url)
        
        if not 'close' in df.columns:
            print('! Failed loading data from AlphaVantage url %s: '%url)
            print('Columns: ' + str(df.columns.values))
            raise FileNotFoundError('Failed to retrieve data from AlphaVantage. You may have exceeded the free 5/min limit. %s'%url)
        else:
            # Make a 'csv' directory
            if not os.path.isdir('csv'):
                os.mkdir('csv')
                
            # Save to file
            df.to_csv(file_path, index=False)
    else:
        # print('Loading data from csv, original url : %s' % url)
        df = pd.read_csv(file_path)
        
    return df

Download multiple months¶

In the query_params above, the months parameter is a list of months in YYYY-MM format which corresponds to the month parameter in AlphaVantage's REST API. Since only one month can be downloaded at a time, we need to split the months up and download each separately, then combine them into one dataframe.

The function download_files requests CSVs as DataFrames multiple times and combines them into a list.

  1. Firstly, it uses urlencode to encode the dictionary of parameters, minus months(which is not an actual API parameter) into a base URL encoding leaving out the month: e.g. https://www.alphavantage.co/query?function=TIME_SERIES_INTRADAY&symbol=SPY... etc.
  2. Then, for each month, it adds the month parameter to the base URL: &month=YYYY-MM and downloads the CSV, then adds it to a list.
  3. Finally, it uses pd.concat(dfs, ignore_index=True) to concatenate the list of DataFrames into a single one. ignore_index serves to drop the original index of each dataframe and create a new index in sequential order. Ignoring the index is necessary because otherwise, the original indices of the respective DataFrames would be maintained such that we had multiple rows with index 0, multiple rows with index 1, etc.

Note: In step 2, we use tqdm to display a progress bar just for quality of life.

# Encoding URLs for downloading data
from urllib.parse import urlencode

# DataFrame manipulation
import pandas as pd

# Progress bar for QOL
from tqdm import tqdm

def download_files(query_params, months=[None]):
    if 'apikey' not in query_params:
        raise Exception("'apikey' key must be present in alphavantage parameters.")
    
    # Leave out months since it's not a real parameter
    new_query_params = query_params.copy()
    del new_query_params['months']

    url = "https://www.alphavantage.co/query?"
    url += urlencode(new_query_params)
    
    dfs = []
    
    # For each slice, download a csv
    for month in tqdm(months, "Downloading from AlphaVantage", ncols=80):
        slice_str = "&month=" + month
        
        new_url = url + slice_str
    
        df = download_csv(new_url, month)
        dfs.append(df)
        
    return pd.concat(dfs, ignore_index=True)

Data Processing¶

Data cleaning¶

The dataframe from above needs to be cleaned a bit. The rows are out of order. Let's sort the rows in ascending order so that the last rows are the most recent. We chose ascending order because if you sample a consecutive sequence of rows for machine learning, they will be in chronological order.

Also, the timestamp column contains datetimes in a text format. Even though the datetime format is such that it can be sorted in alphanumeric order and yield chronological order, we should convert it to a datetime datatype using library functions so we can access properties such as the hour, minute, etc. later without having to do annoying parsing manually. In order to do this, we can use Pandas' datetime functions to convert our text-based timestamp column to np.datetime64 so we can sort the rows properly. We also want to filter out all but standard trading hours, so remove any rows not between 9:30AM and 4:00PM inclusive. Even with extended_hours set to false, AlphaVantage still includes 4:00AM to 8:00PM.

The function clean_dataframe below cleans the dataframe as specified above.

  1. Firstly, it converts the timestamp column to np.datetime64 using pd.to_datetime.
  2. Then, it sorts the dataframe by timestamp using df.sort_values(by="timestamp"). The by parameter specifies which column to sort by.
  3. Then, it drops rows not between 9:30AM and 4:00PM inclusive.
  4. Finally, it resets the index to make sure that our rows are indexed properly in sequential order.
In [ ]:
def clean_dataframe(df):
    # Convert timestamps to np.datetime64
    df.loc[:, 'timestamp'] = pd.to_datetime(df['timestamp'])
    
    # Sort by timestamp in ascending order
    df = df.sort_values(by="timestamp")
        
    start_time = pd.to_datetime('09:30:00').time()
    end_time = pd.to_datetime('16:00:00').time()

    # Filter the DataFrame
    df = df[df['timestamp'].dt.time.between(start_time, end_time)]

    # Reset index and drop old
    df = df.reset_index(drop=True)
    
    return df

Data Processing¶

Let's extract some more information from our data.

For starters, the volume column looks hard to use for machine learning, even if we normalize it. There is massive variation in the column, from near 0 to in the millions. Let's take the log base 10 of the column and call it log_volume.

It also might be hard to predict the direction of price movement from the price columns we see. It would be easier to predict if the columns were mean-centered. For this, let's make a column containing the change in close prices and call it close_delta. We should be careful that each delta is the difference between the current closing price and the last - such that all the columns we have are a priori data, since it doesn't make sense for our model to use data from the future.

We can also add datetime information to our dataframe - let's go with hour, minute, and day of the week.

Finally, we can add things called 'indicators', which are formulas involving moving averages which yield a numerical value that can give a trader information about the momentum of the market. Let's go with two simple moving averages with 20 and 50 intervals, close_sma_20 and close_sma_50 and an exponential moving average with 26 intervals, close_ema_26.

We can analyze these columns later for how much variation they explain before we go on to include them in our model, but for now let's just include them.

Below, we'll define data processing functions, and give a brief overview of each function. The below lists contain parameters for the indicators we'll use, as well as the names of datetime columns we'll use. These lists will be used later - along with the corresponding data processing functions as we define our model.

Simple Moving Average - [Investopedia]¶

$$\text{SMA} = \frac{A_1 + A_2 + \dots + A_n}{n}$$ where $A_n$ is the closing price at period n, and $n$ is the number of total periods.

The simple moving average is simply a mean of the past n periods. It smooths out variability in the closing prices and can quickly be looked at to determine if an asset is in an uptrend(SMA increasing) or a downtrend(SMA decreasing). However, the SMA is a lagging indicator, which means that it provides a signal after the trend begins, so it is not very useful for real-time signals. When the price crosses above the SMA, it could be a buy signal, and vice versa. When a shorter-term SMA crosses above a longer term SMA, this is called a 'golden cross' and typically indicates an uptrend. Conversely, a crossover in the opposite direction suggests a downtrend and is called a 'death cross'.

For more details or more advanced uses of the SMA, check out the Investopedia link in the subtitle.

Important detail: Indicators require multiple datapoints, so we'll have to drop rows from the start. Therefore, in addition to the series we generate from the indicator functions below, we'll also return the highest row index from the original dataset which can be included such that all columns have values. This will be the case for every indicator.

The below function uses Pandas' rolling function to get the sum of each rolling window. The first row which will not contain a null value for the SMA will be n-1.

In [ ]:
def calculate_sma(data, n):
    return data.rolling(n).mean(), n-1
Exponential Moving Average - [Investopedia]¶

$$\text{EMA}_n = A_n (\frac{s}{1+n}) + \text{EMA}_{n-1}(1 - \frac{s}{1+n})$$ where $\text{EMA}_n$ and $\text{EMA}_{n-1}$ are the current and previous EMA respectively, $A_n$ is the current closing price, $s$ is the smoothing factor, and $n$ is the number of total periods. Typically, the smoothing factor is 2.

The exponential moving average at each timestep is a weighted average between the current price and the previous EMA value. It has a similar use to the simple moving average, however, it gives the most recent prices more weight. This means that it responds more quickly than the SMA.

When calculating the EMA without a previous EMA, an SMA with the same period count or the current value can be substituted for the previous EMA.

The below function uses Pandas' ewm function to get the sum of each rolling window with exponential weights. The first row index which will not contain a null value for the EMA will be n-1.

In [ ]:
def calculate_ema(data, n):
    return data.ewm(span=n, adjust=False).mean(), n-1

Change in Closing Price¶

$$\delta = A_n - A_{n-1}$$ This is simply calculated as the difference between the current and previous closing price.

The below function uses Pandas' diff function to get a series containing these differences. The first row index which will not contain a null value for the change in closing price will be 1.

In [ ]:
def calculate_delta(data, n):
    return data.diff(), 1

Log Volume¶

$$\log_{10}(V)$$ This is the logarithm (base 10) of the trading volume, which should make the data easier to use in a machine learning model.

The below function uses Numpy's log10 function to get a series containing the values of the original series with log10 applied.

In [ ]:
def calculate_log_volume(data, n):
    return np.log10(data), 0

Checking influence of various datetime properties on distribution of close delta¶

Let's plot the data distribution, categorized by our various selected datetime properties, using the seaborn violinplot.

It seems like the time of day (in hours) affects the variance, especially at the start and end of the standard trading hours. The variance takes a U shape, starting high, then getting lowest at about noon, then increases again at the end of the day. The minutes also seem to suggest slight differences depending on the time of day. The peak at 30 minutes is likely due to people scrambling to buy/sell as soon as the regular market opens at 9:30. These data seem to be useful indicators of variance, so they could be useful for predictions in our model.

On the other hand, the distribution of the close deltas categorized by day of the week seem to have different extremes, but are otherwise similar. These differences just by looking at them seem like they could be due to random chance or significant events happening on certain days skewing the data.
